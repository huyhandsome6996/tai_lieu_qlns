🧠 Bài 21: Chi tiết _tt / _ui / _dk liên kết thế nào

Bạn đã biết mỗi folder cnXX_* chứa 3 file: _tt.py (thuật toán), _ui.py (giao diện), _dk.py (điều khiển). Nhưng qua câu lệnh nào, file nào chứa câu lệnh đó để 3 file liên kết được với nhau? Bài này sẽ đi từng dòng.

📖 Mục tiêu bài học

Sau bài này bạn sẽ:

  • Hiểu chính xác dòng nào trong _dk.py kéo _ui.py vào.
  • Hiểu chính xác dòng nào trong _dk.py kéo _tt.py vào.
  • Hiểu _ui.py không bao giờ gọi _tt.py — vì sao?
  • Biết luồng dữ liệu 1 chiều: UI → Điều khiển → Thuật toán → (trả kết quả ngược lại).
  • Phân tích 2 ví dụ thực tế: cn01 (Đăng nhập) + cn03 (Thêm sản phẩm).

1. Vì sao chia 3 file? — Nguyên lý tách bạch

Thông thường người mới viết code tất cả trong 1 file: vừa vẽ widget, vừa xử lý logic, vừa gọi DB. Code chạy được nhưng khó bảo trì:

  • Sửa giao diện → sợ vô tình hỏng logic.
  • Test logic → phải khởi tạo GUI (chậm).
  • 2 người làm chung 1 file → conflict liên tục.

Pattern MVC (Model-View-Controller) chia code thành 3 phần độc lập. Đồ án này dùng phiên bản đơn giản hóa của MVC — gọi là pattern _tt/_ui/_dk:

File Vai trò Tương đương MVC Được phép KHÔNG được phép
_tt.py Thuật toán + thao tác dữ liệu Model Import cấu trúc dữ liệu (DSLK, Stack, Queue, SQLite) Import tkinter
_ui.py Vẽ widget Tkinter View Import tkinter, vẽ widget, lưu reference Gọi hàm thuật toán, truy cập DB
_dk.py Điều phối luồng Controller Import cả _ui và _tt, gắn sự kiện Vẽ widget trực tiếp (phải gọi qua _ui)
⚠️ Quy tắc vàng

_ui.py KHÔNG BAO GIỜ import _tt.py — vì UI chỉ vẽ giao diện, không biết logic.

_tt.py KHÔNG BAO GIỜ import _ui.py — vì thuật toán không cần biết widget.

_dk.py IMPORT CẢ HAI — vì điều khiển phải "biết" cả UI (để đọc/ghi) và Thuật toán (để gọi).

Đây là chiều phụ thuộc 1 chiều: _dk → _ui + _tt. Không bao giờ ngược lại.

2. Sơ đồ liên kết 3 file

                        ┌──────────────────────────────┐
                        │       _dk.py (Điều khiển)     │
                        │  • import _ui                 │
                        │  • import _tt                 │
                        │  • tạo UI + Thuật toán        │
                        │  • gắn sự kiện cho nút UI     │
                        └──────────┬──────────┬─────────┘
                                   │          │
                       import      │          │  import
                                   ▼          ▼
        ┌──────────────────────────┐  ┌────────────────────────────┐
        │      _ui.py (Giao diện)   │  │    _tt.py (Thuật toán)     │
        │  • class CuaSoXXXUI       │  │  • class BoXXX hoặc hàm    │
        │  • vẽ widget              │  │  • xử lý DSLK/Stack/Queue  │
        │  • lưu reference widget   │  │  • truy cập SQLite         │
        │  • có hàm getter/setter   │  │  • trả về (bool, msg)      │
        └──────────────────────────┘  └────────────────────────────┘
                       ▲                          ▲
                       │                          │
                       └────────  không import lẫn nhau  ──────────┘

    Chiều dữ liệu khi user bấm nút:

        User click nút
              ↓
        Tkinter gọi hàm trong _dk.py
              ↓
        _dk.py gọi _ui.lay_du_lieu()    ← ĐỌC từ UI
              ↓
        _dk.py gọi _tt.thuc_hien()      ← GỌI thuật toán
              ↓
        _tt trả về (True, msg)
              ↓
        _dk.py gọi _ui.bao_thanh_cong() ← GHI vào UI

3. Ai import ai? — Bảng tổng kết

File nguồn File đích Cho phép? Lý do
_dk.py _ui.py ✓ CÓ Điều khiển cần tạo UI + đọc dữ liệu từ UI
_dk.py _tt.py ✓ CÓ Điều khiển cần gọi thuật toán để xử lý
_ui.py _tt.py ✗ KHÔNG UI không biết logic, chỉ vẽ widget
_ui.py _dk.py ✗ KHÔNG Sẽ tạo phụ thuộc vòng
_tt.py _ui.py ✗ KHÔNG Thuật toán không cần widget
_tt.py _dk.py ✗ KHÔNG Sẽ tạo phụ thuộc vòng
_ui.py tkinter ✓ CÓ UI cần tkinter để vẽ
_tt.py loi_thuat_toan.* ✓ CÓ Thuật toán cần DSLK/Stack/Queue/SQLite

4. Ví dụ 1: cn01 Đăng nhập

Xem chi tiết từng dòng import trong 3 file của cn01:

4.1. File dang_nhap_ui.py — KHÔNG import _tt, KHÔNG import _dk

"""Giao diện form đăng nhập — chỉ vẽ widget, không chứa logic."""

import tkinter as tk
from tkinter import ttk, messagebox
# ↑ Chỉ import tkinter + submodule. KHÔNG import _tt hay _dk.

class CuaSoDangNhapUI:
    def __init__(self, parent):
        self.top = tk.Toplevel(parent)
        # ... vẽ widget ...
        self.nguoi_dung = None  # ← sẽ được _dk gán khi đăng nhập OK

    # Getter cho _dk gọi:
    def lay_ten_dang_nhap(self):
        return self.entry_ten.get().strip()

    def lay_mat_khau(self):
        return self.entry_mk.get().strip()

    def xoa_mat_khau(self):
        self.entry_mk.delete(0, tk.END)

    def bao_loi(self, thong_bao):
        messagebox.showerror('Đăng nhập thất bại', thong_bao, parent=self.top)

    def dong(self):
        self.top.destroy()

Đặc điểm quan trọng: _ui.py có nhiều hàm getter/setter (lay_*, bao_loi, xoa_mat_khau, dong) — đây là "cửa" để _dk.py giao tiếp với UI mà không cần biết widget nào bên trong.

4.2. File dang_nhap_tt.py — KHÔNG import _ui, KHÔNG import _dk

"""Thuật toán đăng nhập — không có code giao diện."""

# Import hàm kiểm tra từ thư viện thuật toán (KHÔNG phải _ui hay _dk)
from loi_thuat_toan.nguoi_dung import kiem_tra_dang_nhap

class BoDangNhap:
    def __init__(self):
        self.nguoi_dung_hien_tai = None

    def dang_nhap(self, ten_dang_nhap, mat_khau):
        """Trả về (True, nguoi_dung) hoặc (False, msg)."""
        try:
            nguoi_dung = kiem_tra_dang_nhap(ten_dang_nhap, mat_khau)
            if nguoi_dung is None:
                return False, 'Sai tên đăng nhập hoặc mật khẩu.'
            self.nguoi_dung_hien_tai = nguoi_dung
            return True, nguoi_dung
        except Exception as loi:
            return False, f'Lỗi: {loi}'

Đặc điểm quan trọng: _tt.py chỉ nhận tham số đầu vào (ten_dang_nhap, mat_khau) và trả về tuple (bool, msg). Không hề biết tới widget nào cả. Vì vậy bạn có thể test _tt.py trong terminal mà không cần mở cửa sổ:

>>> from chuc_nang.cn01_dang_nhap_phan_quyen.dang_nhap_tt import BoDangNhap
>>> bo = BoDangNhap()
>>> bo.dang_nhap('admin', '123')
(True, )
>>> bo.dang_nhap('admin', 'sai_mat_khau')
(False, 'Sai tên đăng nhập hoặc mật khẩu.')

4.3. File dang_nhap_dk.py — IMPORT CẢ _ui VÀ _tt

"""Điều khiển — cầu nối giữa _ui và _tt."""

# ← DÒNG NÀY kéo _ui vào:
from .dang_nhap_ui import CuaSoDangNhapUI
# ← DÒNG NÀY kéo _tt vào:
from .dang_nhap_tt import BoDangNhap


class DieuKhienDangNhap:
    def __init__(self):
        # Tạo 1 đối tượng Thuật toán — giữ suốt vòng đời app
        self.bo = BoDangNhap()  # ← _dk tạo object _tt

    def hien_form_dang_nhap(self, parent):
        # Tạo 1 object UI — modal form
        form = CuaSoDangNhapUI(parent)  # ← _dk tạo object _ui

        # Gắn sự kiện: nút "Đăng nhập" + phím Enter → hàm xử lý
        form.entry_mk.bind('<Return>', lambda e: self._xu_ly_dang_nhap(form))
        form.btn_dn.config(command=lambda: self._xu_ly_dang_nhap(form))

        # Chờ form đóng
        parent.wait_window(form.top)

        # Trả về người dùng (None nếu user đóng form không login)
        return form.nguoi_dung

    def _xu_ly_dang_nhap(self, form):
        """Hàm nội bộ — được gọi khi user bấm nút hoặc Enter."""
        # 1. ĐỌC dữ liệu từ UI (gọi getter của _ui)
        ten = form.lay_ten_dang_nhap()
        mk = form.lay_mat_khau()

        # 2. Validate cơ bản
        if not ten or not mk:
            form.bao_loi('Vui lòng nhập đầy đủ tên đăng nhập và mật khẩu.')
            return

        # 3. GỌI thuật toán (gọi hàm của _tt)
        ok, ket_qua = self.bo.dang_nhap(ten, mk)

        # 4. XỬ LÝ kết quả — GHI ngược vào UI
        if ok:
            form.nguoi_dung = ket_qua  # ← gán cho _ui để caller đọc
            form.dong()                # ← gọi hàm của _ui
        else:
            form.bao_loi(ket_qua)      # ← gọi hàm của _ui
            form.xoa_mat_khau()        # ← gọi hàm của _ui
✅ 2 dòng import quan trọng nhất

Toàn bộ sự liên kết giữa 3 file nằm trong 2 dòng đầu của dang_nhap_dk.py:

from .dang_nhap_ui import CuaSoDangNhapUI  # ← _dk biết _ui
from .dang_nhap_tt import BoDangNhap       # ← _dk biết _tt

Dấu chấm . trước tên module nghĩa là "tìm trong cùng folder với file hiện tại". Vì 3 file cùng folder cn01_dang_nhap_phan_quyen/, nên dùng import tương đối.

Nếu không dùng dấu chấm, phải viết đường dẫn đầy đủ: from chuc_nang.cn01_dang_nhap_phan_quyen.dang_nhap_ui import CuaSoDangNhapUI — dài hơn nhưng cùng ý nghĩa.

4.4. Hành trình của 1 lần bấm nút "Đăng nhập"

BƯỚC 1: User gõ "admin" vào ô tên, "123" vào ô mật khẩu, bấm nút "Đăng nhập"

BƯỚC 2: Tkinter phát sinh event "click" trên form.btn_dn
         ↓
         Tkinter gọi hàm đã gán ở dòng:
         form.btn_dn.config(command=lambda: self._xu_ly_dang_nhap(form))

BƯỚC 3: _dk._xu_ly_dang_nhap(form) chạy
         ↓
         Gọi form.lay_ten_dang_nhap() → trả về "admin"
         Gọi form.lay_mat_khau()      → trả về "123"
         ↓ (đọc dữ liệu từ UI)

BƯỚC 4: Validate: ten="admin", mk="123" → không rỗng → đi tiếp

BƯỚC 5: Gọi self.bo.dang_nhap("admin", "123")
         ↓ (điều khiển gọi thuật toán)

BƯỚC 6: BoDangNhap.dang_nhap() chạy (trong _tt)
         ↓
         Gọi kiem_tra_dang_nhap("admin", "123")  ← từ loi_thuat_toan.nguoi_dung
         ↓
         Hàm này query SQLite: SELECT * FROM nguoi_dung WHERE ten='admin'
         ↓
         Trả về NguoiDung('admin', '123', 'Quản trị viên', 'admin')

BƯỚC 7: BoDangNhap.dang_nhap() trả về (True, NguoiDung(...))
         ↓ (kết quả ngược về _dk)

BƯỚC 8: _dk._xu_ly_dang_nhap() xử lý kết quả
         ↓
         ok=True → gán form.nguoi_dung = NguoiDung(...)
                   gọi form.dong()  ← _ui đóng cửa sổ modal
         ↓ (ghi kết quả vào UI)

BƯỚC 9: form.top.destroy() → wait_window(form.top) trong hien_form_dang_nhap() trả về
         ↓
         return form.nguoi_dung  ← trả NguoiDung về cho main.py

5. Ví dụ 2: cn03 Thêm sản phẩm

Pattern của cn03 gần giống cn01, nhưng có 1 điểm khác: _tt.py là hàm tự do (không bọc trong class) — để bạn thấy cả 2 cách viết.

5.1. them_san_pham_tt.py — hàm tự do

"""Thuật toán thêm sản phẩm — viết dạng hàm tự do (không class)."""

from loi_thuat_toan.lop_doi_tuong import Sach, TapChi, BaoGiay, LuanVan, BanTao


def them_san_pham(danh_sach, bo_undoredo, du_lieu):
    """Thêm 1 sản phẩm vào DSLK + đẩy vào Stack undo.

    Tham số:
        danh_sach   : DSLK đôi toàn cục.
        bo_undoredo : dict 2 Stack.
        du_lieu     : dict chứa thông tin sản phẩm.

    Trả về (True, msg) hoặc (False, msg).
    """
    # 1. Validate
    if not du_lieu.get('ma_so') or not du_lieu.get('ten_san_pham'):
        return False, 'Thiếu mã số hoặc tên sản phẩm.'

    # 2. Kiểm tra mã đã tồn tại chưa
    if danh_sach.tim_theo_ma_so(du_lieu['ma_so']) is not None:
        return False, f"Mã {du_lieu['ma_so']} đã tồn tại."

    # 3. Tạo đối tượng theo loại hàng (đa hình)
    loai = du_lieu.get('loai_hang', 'Sach').lower()
    if loai == 'sach':
        sp = Sach(**du_lieu)
    elif loai == 'tap chi':
        sp = TapChi(**du_lieu)
    elif loai == 'bao giay':
        sp = BaoGiay(**du_lieu)
    elif loai == 'luan van':
        sp = LuanVan(**du_lieu)
    else:
        sp = BanTao(**du_lieu)

    # 4. Thêm vào DSLK
    danh_sach.them_vao_cuoi(sp)

    # 5. Đẩy thao tác vào Stack undo
    bo_undoredo['undo'].day_vao({
        'loai': 'them',
        'ma_so': sp.ma_so,
        'du_lieu_cu': None,
        'du_lieu_moi': du_lieu
    })
    # Xóa redo stack (vì có thao tác mới → redo cũ không còn hợp lệ)
    bo_undoredo['redo'].xoa_tat_ca()

    return True, f'Đã thêm "{sp.ten_san_pham}" thành công.'

5.2. them_san_pham_ui.py — vẽ form Thêm

UI của Thêm/Sửa dùng chung 1 file trong _shared/form_them_sua_ui.py. UI chỉ vẽ widget + có hàm lay_du_lieu() để dk đọc:

"""Form Thêm/Sửa dùng chung — chỉ vẽ widget."""

import tkinter as tk
from tkinter import ttk


class FormThemSuaUI:
    def __init__(self, parent, du_lieu_cu=None):
        """Nếu du_lieu_cu=None → chế độ Thêm. Nếu có → chế độ Sửa (điền sẵn)."""
        self.top = tk.Toplevel(parent)
        self.top.title('Thêm sản phẩm' if du_lieu_cu is None else 'Sửa sản phẩm')
        # ... modal setup ...

        self.ket_qua = None  # ← _dk sẽ gán khi Lưu thành công

        # Vẽ các widget: Entry mã số, tên, Combobox loại, Entry giá, ...
        self._tao_giao_dien(du_lieu_cu)

    def _tao_giao_dien(self, du_lieu_cu):
        # ... vẽ Label + Entry + Combobox + Button Lưu/Hủy ...
        self.btn_luu = ttk.Button(self.top, text='Lưu')
        # ← KHÔNG gắn command ở đây. _dk sẽ gắn.
        self.btn_huy = ttk.Button(self.top, text='Hủy')
        # ← Tương tự.

    # Getter cho _dk đọc:
    def lay_du_lieu(self):
        return {
            'ma_so': self.entry_ma.get().strip(),
            'ten_san_pham': self.entry_ten.get().strip(),
            'loai_hang': self.combo_loai.get(),
            'gia_co_ban': int(self.entry_gia.get() or 0),
            'ton_kho': int(self.entry_ton.get() or 0),
            # ...
        }

    def bao_loi(self, msg):
        messagebox.showerror('Lỗi', msg, parent=self.top)

    def bao_thanh_cong(self, msg):
        messagebox.showinfo('Thành công', msg, parent=self.top)

    def dong(self):
        self.top.destroy()

5.3. them_san_pham_dk.py — liên kết _ui và _tt

"""Điều khiển — cầu nối UI và Thuật toán."""

# ← IMPORT _tt (hàm tự do, không phải class)
from .them_san_pham_tt import them_san_pham
# ← IMPORT _ui (form Thêm/Sửa dùng chung)
from .them_san_pham_ui import FormThemSuaUI


class DieuKhienThemSanPham:
    def __init__(self, danh_sach, bo_undoredo):
        """Nhận state từ QuanLyTrungTam khi được tạo ở cua_so_chinh_dk."""
        self.danh_sach = danh_sach
        self.bo_undoredo = bo_undoredo

    def mo_form_them(self, parent):
        """Mở form Thêm modal. Trả True nếu đã thêm, False nếu user hủy."""
        # 1. Tạo UI
        form = FormThemSuaUI(parent, du_lieu_cu=None)

        # 2. Gắn sự kiện nút Lưu → hàm xử lý
        form.btn_luu.config(command=lambda: self._xu_ly_luu(form))

        # 3. Chờ form đóng
        parent.wait_window(form.top)

        # 4. Trả về dựa trên form.ket_qua (được gán trong _xu_ly_luu)
        return form.ket_qua is not None

    def _xu_ly_luu(self, form):
        """Hàm nội bộ — gọi khi user bấm nút Lưu."""
        # 1. ĐỌC dữ liệu từ UI
        du_lieu = form.lay_du_lieu()

        # 2. GỌI thuật toán
        ok, msg = them_san_pham(self.danh_sach, self.bo_undoredo, du_lieu)

        # 3. XỬ LÝ kết quả
        if ok:
            form.ket_qua = du_lieu       # ← gán để mo_form_them biết
            form.bao_thanh_cong(msg)     # ← GHI vào UI
            form.dong()                  # ← GHI vào UI (đóng form)
        else:
            form.bao_loi(msg)            # ← GHI vào UI (giữ form để sửa)

6. Luồng dữ liệu 1 chiều

Qua 2 ví dụ trên, bạn thấy luồng dữ liệu luôn đi theo 1 chiều:

        ┌─────────────────────────────────────────────────────┐
        │                     USER                            │
        │  (gõ text, bấm nút, double-click, nhấn phím)         │
        └──────────────────────┬──────────────────────────────┘
                               │
                               ▼ event
        ┌─────────────────────────────────────────────────────┐
        │                  _ui.py (WIDGET)                     │
        │  ← Tkinter gọi callback đã gắn                       │
        └──────────────────────┬──────────────────────────────┘
                               │
                               ▼ gọi hàm _dk
        ┌─────────────────────────────────────────────────────┐
        │              _dk.py (ĐIỀU KHIỂN)                     │
        │  1. form.lay_du_lieu()  ← ĐỌC UI                     │
        │  2. self.bo.xxx()      ← GỌI _tt                     │
        └──────────────────────┬──────────────────────────────┘
                               │
                               ▼ gọi hàm _tt
        ┌─────────────────────────────────────────────────────┐
        │              _tt.py (THUẬT TOÁN)                     │
        │  • thao tác DSLK / Stack / Queue                     │
        │  • query SQLite                                      │
        │  • trả về (True, msg) hoặc (False, msg)              │
        └──────────────────────┬──────────────────────────────┘
                               │
                               ▼ return

💡 Tại sao 1 chiều?

Vì nếu 2 chiều (UI gọi _tt trực tiếp), UI sẽ phải import _tt → phụ thuộc lẫn nhau → khó test riêng.

Với 1 chiều, bạn có thể test từng phần độc lập:

  • Test _tt.py trong terminal: python -c "from cn03_them_san_pham_tt import them_san_pham; ..."
  • Test _ui.py bằng cách mở form + gõ thử + kiểm tra getter trả đúng.
  • Test _dk.py bằng cách mock _ui và _tt (giả lập trả về).

7. Cú pháp bind sự kiện Tkinter — chi tiết

Trong _dk.py luôn có dòng gắn sự kiện dạng:

# Cách 1: Gắn command cho Button
form.btn_dn.config(command=lambda: self._xu_ly_dang_nhap(form))

# Cách 2: Gắn phím cho Entry
form.entry_mk.bind('<Return>', lambda e: self._xu_ly_dang_nhap(form))

# Cách 3: Gắn double-click cho Treeview
self.ui_bang.tree.bind('<Double-1>', lambda e: self._sua_san_pham())

7.1. button.config(command=...)

Widget ttk.Button có tham số command — gán hàm sẽ chạy khi user bấm. Hàm KHÔNG nhận tham số (vì Tkinter gọi hàm mà không truyền gì).

# Hàm xử lý:
def _xu_ly_dang_nhap(form):
    ten = form.lay_ten_dang_nhap()
    # ...

# Gắn cho nút:
form.btn_dn.config(command=lambda: _xu_ly_dang_nhap(form))
#                ↑ lambda bọc vì command cần hàm 0 tham số, nhưng _xu_ly cần tham số form

7.2. widget.bind('<Event>', handler)

Widget bất kỳ đều có .bind(event, handler). Handler PHẢI nhận 1 tham số event (Tkinter tự truyền).

Các event phổ biến trong đồ án:

Event Ý nghĩa Dùng trong cn nào
<Return> Phím Enter cn01 (Đăng nhập), cn06 (Tìm kiếm)
<Double-1> Double-click chuột trái cn02 (bảng) → mở cn04 (Sửa)
<Button-1> Click chuột trái (1 lần) Hiếm dùng trong đồ án
<Key> Bấm phím bất kỳ cn03 (form Thêm) — để validate
<ComboboxSelected> Chọn item trong Combobox cn03, cn04 (chọn loại hàng), cn06 (chọn tiêu chí sắp xếp)

7.3. Tại sao dùng lambda?

Vì khi gán command=_xu_ly_dang_nhap(form) — Python sẽ gọi hàm ngay lập tức và gán kết quả trả về. Bạn muốn gán hàm chưa gọi để Tkinter gọi sau. Dùng lambda tạo hàm vô danh chưa gọi:

# SAI: Python gọi _xu_ly ngay khi gán (form chưa có dữ liệu!)
form.btn_dn.config(command=_xu_ly_dang_nhap(form))

# ĐÚNG: lambda tạo hàm 0 tham số, Tkinter sẽ gọi sau khi user bấm
form.btn_dn.config(command=lambda: _xu_ly_dang_nhap(form))

# ĐÚNG (cho bind, handler phải nhận event):
form.entry_mk.bind('<Return>', lambda e: _xu_ly_dang_nhap(form))
#                                       ↑ 'e' là event object, bỏ qua không dùng

8. 5 quy tắc pattern để nhớ

✅ Ghi nhớ 5 quy tắc này để đọc code không rối
  1. _dk import _ui + _tt. 2 dòng đầu mỗi file _dk.py luôn là 2 import này (dùng dấu chấm .).
  2. _ui chỉ import tkinter. UI chỉ vẽ widget + có getter/setter. KHÔNG gọi thuật toán.
  3. _tt chỉ import loi_thuat_toan.*. Thuật toán chỉ nhận tham số + trả tuple. KHÔNG import tkinter.
  4. Luồng 1 chiều: UI → dk (đọc) → tt (gọi) → dk (ghi) → UI.
  5. Event handler dùng lambda: btn.config(command=lambda: ham()) cho nút, widget.bind('<Event>', lambda e: ham()) cho sự kiện.
