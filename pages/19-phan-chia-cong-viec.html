2. Chi tiết Huy (9 phần)

Hồ Quang Huy — mạnh nhất, phụ trách thuật toán khó

Nguyên tắc: Phụ trách những phần cần hiểu sâu về cấu trúc dữ liệu và thuật toán — DSLK đôi + Merge Sort O(n log n). Huy là người duy nhất có khả năng cài đặt Merge Sort đệ quy trên DSLK.

9 phần cụ thể

  1. DSLK đôi — Lớp Node + DoublyLinkedList với CRUD: them_vao_cuoi, xoa_node, cap_nhat_node, tim_theo_ma_so, tim_kiem, tim_theo_the_loai, tim_kiem_nang_cao.
  2. Merge Sort — Hàm sap_xep_tron (đệ quy) + các hàm phụ trợ: _chia_doi (Fast/Slow pointer), _tron (trộn 2 DSLK đã sắp), _lay_gia_tri, _dem_node.
  3. Thống kê trong DSLK — Hàm thong_ke() duyệt 1 lần, tích lũy 7 chỉ số.
  4. CN02 — Đọc danh sách — Hàm lay_danh_sach + BangSanPhamUI (Treeview) + DieuKhienDocDanhSach.
  5. CN06 — Tìm kiếm + Sắp xếp — Thanh UI + 2 hàm tim_kiem + sap_xep (gọi Merge Sort của DSLK).
  6. CN07 — Thống kê — KhungThongKeUI (5 label) + DieuKhienThongKe (cập nhật label từ dict thong_ke).
  7. Khung chính — khung_chinh_tt.py (QuanLyTrungTam) + cua_so_chinh_ui.py + cua_so_chinh_dk.py (điều phối 8 điều khiển con).
  8. main.py — Điểm vào ứng dụng: tạo QLTT + Tk root + form đăng nhập → mở cửa sổ chính.
  9. Tích hợp + Debug — Ghép code 3 người, chạy test, sửa bug.

3. Chi tiết Ngọ (9 phần)

Bùi Gia Ngọ — đã học SQLite + Stack + Queue

Nguyên tắc: Phụ trách phần dữ liệu bền vững (SQLite) và 2 cấu trúc dữ liệu tự cài (Stack LIFO + Queue FIFO). Đây là phần Ngọ đã học, dễ tiếp thu.

9 phần cụ thể

  1. Stack LIFO (NganXep) — Lớp NodeNganXep + NganXep với: day_vao, lay_ra, xem_dinh, rong, xoa_tat_ca.
  2. Queue FIFO (HangDoi) — Lớp NodeHangDoi + HangDoi với: them_vao, lay_ra, xem_danh_sach, rong, xoa_tat_ca, xoa_theo_ma.
  3. SQLite — kết nối + tạo bảng — lay_duong_dan_db, tao_bang_neu_chua_co (2 bảng: mat_hang + nguoi_dung).
  4. SQLite — CRUD sản phẩm — luu_mot_san_pham, cap_nhat_san_pham, xoa_mot_san_pham, tai_du_lieu, tao_du_lieu_mau_neu_rong, xuat_du_lieu_csv.
  5. SQLite — người dùng — tao_nguoi_dung, xac_thuc_nguoi_dung, co_nguoi_dung_nao_chua.
  6. NguoiDung + phân quyền — Lớp NguoiDung (ho_ten, vai_tro, la_admin), hàm kiem_tra_dang_nhap, khoi_tao_nguoi_dung_mac_dinh.
  7. CN01 — Đăng nhập + phân quyền — BoDangNhap + CuaSoDangNhapUI (modal) + DieuKhienDangNhap.
  8. CN08 — Undo/Redo — 3 hàm hoan_tac, lam_lai, trang_thai; đảo ngược 3 loại thao tác them/sua/xoa; _tao_doi_tuong_tu_dict.
  9. CN09 — Giỏ hàng + thanh toán — 4 hàm them_vao_gio, xem_gio_hang, xoa_khoi_gio, thanh_toan; KhungGioHangUI; DieuKhienGioHang.

4. Chi tiết Tú (9 phần)

Phan Anh Tú — chỉ học OOP, làm phần OOP + CRUD đơn giản

Nguyên tắc: Tú chỉ mới học OOP, nên phụ trách phần OOP (lớp cha + 5 lớp con kế thừa) và 3 chức năng CRUD (Thêm/Sửa/Xóa) — đây là phần dễ nhất về thuật toán, chỉ cần hiểu OOP cơ bản.

9 phần cụ thể

  1. MatHang (lớp cha trừu tượng) — ABC + abstractmethod, các thuộc tính chung (ma_so, ten, gia_co_ban, ton_kho), hàm trừu tượng tinh_gia_ban.
  2. Sach — Lớp con kế thừa, thêm tac_gia + nha_xuat_ban, override tinh_gia_ban (×1.1), override chuyen_thanh_dict.
  3. TapChi — Lớp con, thêm so_phat_hanh, tinh_gia_ban ×1.05.
  4. BaoGiay — Lớp con, thêm ngay_xuat_ban, tinh_gia_ban ×1.0.
  5. LuanVan — Lớp con, thêm tac_gia + truong_dai_hoc + nam_bao_ve, tinh_gia_ban ×1.2.
  6. BanThao — Lớp con, thêm tac_gia + tinh_trang, tinh_gia_ban ×1.15.
  7. CN03 — Thêm sản phẩm — Hàm them_san_pham (đa hình) + điều khiển + re-export FormThemSuaUI.
  8. CN04 — Sửa sản phẩm — Hàm sua_san_pham + DieuKhienSuaSanPham (tìm sp → mở form Sửa với dữ liệu cũ).
  9. CN05 — Xóa sản phẩm — Hàm xoa_san_pham (kiểm tra admin) + 3 hàm UI (hoi_xac_nhan_xoa, bao_loi, bao_thanh_cong) + DieuKhienXoaSanPham.
💡 Form Thêm/Sửa dùng chung

Vì form Thêm và form Sửa gần như giống nhau, Tú phụ trách file chuc_nang/_shared/form_them_sua_ui.py — dùng chung cho cả CN03 và CN04.

Đây là pattern tái sử dụng code — tránh lặp, dễ bảo trì.

6. Điểm liên kết giữa 3 người

Các phần phụ thuộc lẫn nhau — cần thống nhất giao diện hàm trước khi bắt đầu:

6.1. Tú → Huy + Ngọ

Tú làm OOP trước, vì Huy (DSLK) và Ngọ (CN01-09) đều cần MatHang và 5 lớp con. Phải thống nhất:

  • Tên lớp + tên thuộc tính (ma_so, ten_san_pham, ...).
  • Chữ ký hàm __init__ + tinh_gia_ban + chuyen_thanh_dict.

6.2. Huy → Ngọ

Huy làm DSLK + Merge Sort trước. Ngọ cần dùng DSLK trong các chức năng cn01/cn08/cn09. Thống nhất:

  • API DSLK: them_vao_cuoi, xoa_node, tim_theo_ma_so, cap_nhat_node, chuyen_thanh_danh_sach, thong_ke.

6.3. Ngọ ↔ Huy

Ngọ làm Stack/Queue + SQLite, Huy cần dùng trong cn02/cn06/cn07. Thống nhất:

  • API Stack: day_vao, lay_ra, rong, xoa_tat_ca.
  • API Queue: them_vao, lay_ra, xem_danh_sach, xoa_theo_ma.
  • API SQLite: luu_mot_san_pham, cap_nhat_san_pham, xoa_mot_san_pham, tai_du_lieu.

7. Hướng dẫn thuyết trình

Khi bảo vệ đồ án, mỗi người thuyết trình phần mình làm. Thứ tự đề xuất:

  1. Huy mở đầu: Giới thiệu đồ án + cấu trúc tổng thể + main.py + QuanLyTrungTam.
  2. Tú tiếp: Giải thích OOP (lớp cha + 5 con + đa hình) + demo CN03/CN04/CN05.
  3. Ngọ tiếp: Giải thích SQLite + Stack/Queue + demo CN01/CN08/CN09.
  4. Huy kết: Giải thích DSLK + Merge Sort + demo CN02/CN06/CN07.
  5. Cả 3: Trả lời câu hỏi + chạy test_app.py cho thấy 47 test PASS.

Mẹo thuyết trình

💡 5 mẹo để thuyết trình tốt
  1. Chuẩn bị slide ngắn gọn: 1 slide mỗi chức năng, screenshot UI.
  2. Demo trực tiếp: Chạy app, thêm/sửa/xóa, undo/redo, giỏ hàng.
  3. Giải thích cú pháp mới: Tkinter Treeview, Combobox, StringVar, lambda.
  4. Nhấn mạnh thuật toán: Merge Sort O(n log n), Stack/Queue tự cài.
  5. Chạy test_app.py: 47 test PASS = bằng chứng code chạy đúng.

Câu hỏi thường gặp khi bảo vệ

Câu hỏi Trả lời ngắn
Tại sao tự cài DSLK mà không dùng list Python? Theo yêu cầu môn Lập trình Nâng cao — hiểu sâu về con trỏ/tham chiếu.
Merge Sort có ưu điểm gì so với Bubble Sort? O(n log n) thay vì O(n²) — nhanh hơn nhiều với danh sách lớn.
Đa hình thể hiện ở đâu? Mỗi lớp con override tinh_gia_ban() với công thức riêng — cùng lời gọi ra kết quả khác nhau.
Tại sao dùng Stack cho Undo/Redo? Vì thao tác gần nhất cần hoàn tác đầu tiên — đúng nguyên lý LIFO.
Tại sao dùng Queue cho Giỏ hàng? Vì món nào vào trước thanh toán trước — đúng nguyên lý FIFO.
Phân quyền admin/nhân viên ở đâu? 2 lớp: UI vô hiệu nút Xóa + thuật toán kiểm tra la_admin.
Làm sao chống SQL Injection? Dùng parameterized query (dấu ? thay vì nối chuỗi).
Tại sao chia 3 file _tt/_ui/_dk? Tách biệt logic / giao diện / điều khiển — dễ bảo trì + test độc lập.
